01

The one-sentence thesis

Don't choose between a rich UI and an autonomous agent — ship both off one contract.

No "AI bolted on the side." The Action is the atom of the system. UI, agent, HTTP, MCP, A2A, and CLI are all just callers of actions. Internalize that and the rest follows. 

// One action powers UI, agent, HTTP, MCP, A2A, and CLI — all of them.
export default defineAction({
  description: "Get a single email by ID, including its full body.",
  schema: z.object({ id: z.string() }),
  http: { method: "GET" },
  run: async ({ id }) => {
    /* read scoped SQL, return data */
  },
});

What this repo actually is

It's the framework plus the showcase apps — not a single product. Four layers worth knowing:

packages/core

@agent-native/core — the runtime

CLI, agent loop, action engine, server plugins, Vite plugin, sync, auth, collab, observability. The published npm package every template depends on. Most "how does the magic work" answers live here.

templates/*

15 cloneable SaaS apps

mail, calendar, slides, analytics, content, plan, clips… Each is a standalone app with its own Drizzle schema, actions/, app/ (React), and server/ (Nitro). Both reference and product.

packages/{dispatch,…}

Satellite packages

Workspace control plane (vault/integrations), scheduling primitives, pinpoint visual feedback, embedding SDK, migration workbench, dev frame.

.agents/skills + skills

The real docs

Prose playbooks the agent reads before touching an area. AGENTS.md/CLAUDE.md are deliberately thin and point here. The Skill Index is your map.

Fork hygiene: origin is your fork (borahanmirzaii/agent-native); upstream is BuilderIO/agent-native. Land work on your fork — leave upstream untouched.

05

The non-negotiable design patterns

These are enforced by pnpm guards and CI. Each guard codifies a real past incident — break a rule and the suite catches it.

  1. Actions are the single source of truth. Define ops in actions/; agent gets tools, frontend uses useActionQuery/useActionMutation. Never hand-write fetch — add a named helper first.
  2. No twin routes. Don't wrap an action in a REST endpoint that re-exports it. Inspect existing actions before adding any custom route for app data.
  3. Data lives in SQL via Drizzle, provider-agnostic. Never assume SQLite. Schema changes are additive only — never drop/rename/truncate, never drizzle-kit push at prod.
  4. Scoped access on ownable data. ownableColumns() tables read/write through accessFilter / resolveAccess / assertAccess. Custom routes establish request context first.
  5. App state lives in application_state (SQL) so the agent always knows current navigation, selection, and focused object.
  6. All AI work goes through the agent chat. UIs never call an LLM directly — delegate to the agent.
  7. Secrets are never hardcoded. Vault/secrets own values; apps own readers. Use workspace integration grants, not copied tokens.
  8. Optimistic UI by default. Update cache + navigate immediately, roll back on error. Spinners only for destructive ops.
  9. The four-area rule. Every feature touches UI · actions · skills/instructions · application state when applicable.
  10. The agent can edit the app's own source. Design UI and data flows to be legible to an agent.
06

How data flows — two concrete traces

Same atoms, same scoping, both directions. That symmetry is the framework.

A · User clicks "Archive" in the UI
human → action → SQL → everyone
Button → useActionMutation("archive-email", {id}); optimistic cache update fires instantly.
Request hits the action surface → archive-email run().
Scoped Drizzle write — access checked via resolveAccess.
Write lands in SQL; application_state reflects new view.
useDbSync() notifies every surface — other tabs, users, and the agent.
B · "Archive everything from Stripe" in chat
agent → same action → SQL → live UI
Agent runtime's tools are the same actions (list-emails, archive-email).
Agent calls list-emails, then archive-email — identical run(), identical scoping.
No separate "AI path." Same guarantees as the click.
SQL write → sync → UI updates live while the agent narrates.
The agent sees the result via application_state, continues.

Daily workflow & commands

pnpm install                 # postinstall builds core + sibling packages
pnpm dev                     # lazy dev (recommended)
pnpm --filter mail dev       # run a single template
pnpm --filter @agent-native/core dev   # watch-build the framework

pnpm run prep                # fmt + typecheck + test + guards — RUN BEFORE PUSH
pnpm typecheck               # all packages + templates
pnpm test                    # core + migrate + docs + dispatch + brain evals
pnpm run guards              # the security/consistency guard suite
env

Database & keys

DATABASE_URL unset → local SQLite at data/app.db. Set it for Postgres/Neon/Supabase/Turso. ANTHROPIC_API_KEY required for agent chat.

releases

Changesets

Source changes in packages/{core,dispatch,scheduling,pinpoint} need a .changeset/*.md. Never bump versions by hand.

safety

Guards = incidents

Read the header of any scripts/guard-*.mjs to see the invariant it protects (cross-tenant leaks, credential leaks, prod push…).

Three product shapes, same primitives

You never rebuild the agent contract to change shape — you wrap more (or less) UI around the same actions.

ShapeYou shipUnderneath
HeadlessCall agent + actions from code / CLI / HTTP / MCP / A2A.actions, auth, jobs
Rich chatStandalone/embedded chat with native tool renderers.shared chat runtime
Whole appFull SaaS UI; chat moves sidebar ↔ center, stays synced.SQL state + actions
Mental shortcut: "Where do I add capability X?" → an action. "How does the agent know about Y?" → it's a tool (action) or it's in application_state. "Why won't the UI update?" → sync wiring, not a render bug, nine times out of ten.

First-week checklist

  • Run one template. pnpm install && pnpm --filter mail dev with local SQLite + ANTHROPIC_API_KEY set.
  • Read one full action top-to-bottom (templates/mail/actions/get-email.ts); trace its scoping + return shape.
  • Watch the symmetry. Archive an email in the UI, then ask the agent to archive one — both hit the same action.
  • Skim the runtime. packages/core/src/agent/ and action.ts to see the loop.
  • Read the 5 core skills: adding-a-feature, actions, storing-data, real-time-sync, security — ~80% of daily work.
  • Learn green. Run pnpm run prep once so you know what passing looks like before you push.
